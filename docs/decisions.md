# Decision log

Lightweight log of decisions that shape the platform. One entry per
decision: date, decision, reason. Newest entries at the bottom.

| Date       | Decision                                                                                                                                                               | Reason                                                                                                                                                                                                                                                                                                                                                       |
| ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 2026-06-11 | Build one multi-tenant SaaS platform rather than bespoke dashboards per client                                                                                         | Bespoke builds take too long per client; one platform with per-client workspaces and per-subscription modules scales across the agency's client base.                                                                                                                                                                                                        |
| 2026-06-11 | Integrate around specialist software clients already rely on (Motion Kinetic, Xero, Zoho, Stripe); never try to replace it                                             | Clients trust their specialist tools; the platform's value is connecting and surfacing, not replacing.                                                                                                                                                                                                                                                       |
| 2026-06-11 | Next.js (latest stable) with TypeScript strict, Tailwind and shadcn/ui                                                                                                 | Mainstream, well-documented stack; strict TypeScript catches errors early; shadcn/ui gives consistent components without a heavy design system.                                                                                                                                                                                                              |
| 2026-06-11 | Supabase for Postgres, Auth, RLS and Storage                                                                                                                           | One managed service covers database, authentication, row-level tenancy enforcement and file storage, keeping the moving parts to a minimum.                                                                                                                                                                                                                  |
| 2026-06-11 | Vercel for hosting                                                                                                                                                     | First-party Next.js hosting with preview deployments and per-environment secrets.                                                                                                                                                                                                                                                                            |
| 2026-06-11 | Resend for email, with a verified sending domain per client                                                                                                            | Clients' outbound email (for example lead notifications) should come from their own domain, not a shared one.                                                                                                                                                                                                                                                |
| 2026-06-11 | Trunk-based development, push direct to main                                                                                                                           | Solo development in small approved passes; branches and PRs add ceremony without review benefit at this stage.                                                                                                                                                                                                                                               |
| 2026-06-11 | Billing is manual at first: BSK invoices clients and sets entitlements in an admin area                                                                                | Avoids building Stripe billing before there are paying tenants; the entitlement model is designed to map onto one future Stripe subscription with multiple line items.                                                                                                                                                                                       |
| 2026-06-11 | Soft delete via a nullable deleted_at timestamp on leads, customers, sites and quotes when those tables are built, with a separate deliberate hard-erase path for GDPR | Clients will delete things by accident and expect recovery, but GDPR erasure must be real deletion, so the two needs get two distinct mechanisms. Organisations and memberships already have status columns that serve as their soft-delete equivalent, so they do not get deleted_at. The column is not added to any table until the table itself is built. |
| 2026-06-13 | Resolve the moderate postcss XSS advisory (GHSA-qx2v-qp2m-jg93, unescaped close-style tag in CSS stringify output, postcss <8.5.10) with an npm overrides pin of postcss to ^8.5.10, deduping next 16.2.9's private postcss 8.4.31 onto the 8.5.15 the app already resolves. The advisory was wrongly attributed to pdf-lib when it was added last pass; pdf-lib's own subtree is clean. | npm's only automatic fix downgrades Next to 9.3.3, a breaking change, so it was refused. The override is a within-8.x patch bump that build, lint, tsc and the full PDF suite all pass unchanged, and npm audit then reports 0 vulnerabilities. The flaw is in postcss's build-time CSS stringify path; our build runs only authored Tailwind CSS through postcss, never untrusted input, so the vulnerable path is not one we exercise. |
| 2026-06-15 | The product's user-facing name is BSK View (with a simple wordmark); the working name Relay is retired. The repository stays bsk-platform. | A clear product name and mark for clients; agreed at the start of the design work. Internal/repo naming is unaffected, so no code or migration churn beyond UI copy and docs. |
| 2026-06-15 | Establish the design system: dark theme as the default with a light theme, a soft cool-neutral canvas, and a single brand accent (a calm indigo, #5b5bd6) driven per workspace by organisations.brand_color through the --brand CSS variable. Theme choice persists in a cookie (per browser). | The product needed a coherent, premium-but-restrained look (the Linear/Stripe register) on top of the function-first build. Driving the accent from one CSS variable means a workspace's brand colour re-themes the app everywhere; a cookie keeps the theme without a schema change, and dark-default suits the calm canvas. The neutral warmth and the default accent shade are tasteful defaults chosen to be steered from the live result. |
